# Cross-Skill Collaboration Protocol: solana-incident-response-skill

This document defines the cross-skill integration boundaries, signals, and post-incident feedback loops for `solana-incident-response-skill` within the 5-skill Solana multi-agent engineering ecosystem.

---

## 1. Inbound Signals & Active Triggers

The Incident Response skill is designed to receive critical alerts from monitoring systems and execute immediate containment triage playbooks.

### A. From `solana-observability-skill`
* **Signal ID:** `OBS_CRITICAL_ANOMALY` / `OBS_ORACLE_SKEW`
* **Trigger Action:**
  1. The receipt of a critical anomaly alert automatically executes the `/incident-triage` command.
  2. Input parameters (such as `triggeringSignature` and `observedValue`) are parsed into the rapid risk score calculator.
  3. If severity is evaluated as P0/P1, the `incident-commander.md` and `forensic-investigator.md` agents are spawned instantly.

---

## 2. Outbound Signals & Payloads

During an active incident or immediately following containment, the Incident Commander emits containment signals to protect other parts of the ecosystem:

### A. To `solana-token-launch-skill`
* **Trigger Event:** Forensic analysis confirms the protocol's token contract, treasury vault, or mint authority is compromised.
* **Outbound Action:** Force-engage token safety rules. Prevent further token generation events, freeze target token accounts, or initiate temporary liquidity pool trading halts.
* **Signal Payload:**
  ```json
  {
    "signal": "INC_TOKEN_COMPROMISED",
    "timestamp": 1719325546,
    "params": {
      "tokenMint": "EPjFW3bdKy91sbq50GjSzTnhF1gXQC2yUtCdB9ksja2",
      "actionRequired": "Engage TGE Safety Rules / Freeze Mint Authority",
      "proofSignature": "3yZp28aSgYd...evidence_signature..."
    }
  }
  ```

### B. To `solana-ux-skill`
* **Trigger Event:** DNS, CDN, or front-end hosting surfaces show signs of hijacking or unauthorized script injection (e.g., wallet draining payloads).
* **Outbound Action:** Instruct the UX engine to inject anti-drain warning modals and temporarily suspend the connect-wallet button interface.
* **Signal Payload:**
  ```json
  {
    "signal": "INC_FRONTEND_COMPROMISED",
    "timestamp": 1719325546,
    "params": {
      "compromisedDomain": "https://protocol-ui.network",
      "remediationFlow": "Load wallet-ux.md Drain Prevention Layouts",
      "suspendInteractions": true
    }
  }
  ```

### C. To `solana-depin-builder-skill`
* **Trigger Event:** On-chain forensics identifies that the exploit leveraged an oracle data discrepancy or oracle key hijack.
* **Outbound Action:** Command the DePIN registry to ignore updates from the compromised oracle and fall back to local PoPW validation thresholds.
* **Signal Payload:**
  ```json
  {
    "signal": "INC_ORACLE_EXPLOIT",
    "timestamp": 1719325546,
    "params": {
      "compromisedOracle": "PythOracleAddress11111111111111111111",
      "programId": "DepnRegistry1111111111111111111111111111",
      "remediationFlow": "Load oracle-integration.md for Hardening"
    }
  }
  ```

### D. To `solana-observability-skill`
* **Trigger Event:** Active containment has been completed, moving from response to the post-incident monitoring phase.
* **Outbound Action:** Force-scale the telemetry checklist to trace any new attempts of the entry vector pattern.
* **Signal Payload:**
  ```json
  {
    "signal": "INC_POST_INCIDENT_MONITORING",
    "timestamp": 1719325546,
    "params": {
      "suspectedPattern": "Instruction Discriminator: 0x3f5c7112",
      "monitorTargets": ["VaultPDA1...", "OraclePDA2..."],
      "alertOnFirstHit": true
    }
  }
  ```

## 3. Post-Incident Feedback Loop & Hardening Protocol

Every post-mortem document generated by the `/post-mortem-template` command must output exactly one concrete hardening action item for each of the other four skills:

1. **For `solana-depin-builder-skill`:**
   - *Action:* Hardened verification constraints on GPS localization coordinates (e.g., require multi-peer signatures before updating device state).
2. **For `solana-observability-skill`:**
   - *Action:* Create custom trace rule dashboards matching the specific instruction discriminators abused in the exploit transaction.
3. **For `solana-token-launch-skill`:**
   - *Action:* Transfer the upgrade authority of the launch pool token contract from single developer keys to a Squads v4 multisig with a 3-of-5 signature threshold.
4. **For `solana-ux-skill`:**
   - *Action:* Revise transaction simulation layouts to explicitly show the expected maximum balance delta before user confirmation.

## 4. Cross-Skill Query Interface & API Templates

Incident Response agents query other skills in the ecosystem using the following structured formats:

### Query to `solana-token-launch-skill`
* **Intent:** Retrieve current signer thresholds for the token treasury vault.
* **Query Format:**
  ```text
  /query-token --action get_vault_threshold --vault_address [VAULT_PDA]
  ```

### Query to `solana-observability-skill`
* **Intent:** Pull all failed transactions for program ID within a 10-slot window.
* **Query Format:**
  ```text
  /query-observability --action get_failed_signatures --program_id [PROGRAM_ID] --slot_range [START_SLOT]-[END_SLOT]
  ```

### Query to `solana-ux-skill`
* **Intent:** Check if a standard warning model is available on the compromised domain.
* **Query Format:**
  ```text
  /query-ux --action verify_drain_prevention --domain_name [DOMAIN] --modal_id "DrainAlertModal"
  ```

### Query to `solana-depin-builder-skill`
* **Intent:** Pull device verification status for suspected sybil nodes.
* **Query Format:**
  ```text
  /query-depin --action check_node_verification --node_pubkey [NODE_PUBKEY]
  ```

## 5. Shared Ecosystem Vocabulary

- **Epoch:** A slot-based timeframe (standard: 432,000 slots) used to structure timeline evidence and trace historical outflows.
- **Proof:** Cryptographic signatures or account states that serve as hard evidence of the exploit or containment.
- **Stake:** Collateral locks inspected by the forensic investigator to evaluate financial exposure.
- **Oracle:** The state reporting component (Pyth/Switchboard) whose manipulation triggered the incident-triage.
- **Node:** An RPC node or gateway used to fetch transaction logs and logs traces.
- **Commitment:** The transaction finality state (e.g., check that a pause transaction has reached `finalized` commitment before exiting P0).

- [ ] Confirm all outbound hardening signals were received and acknowledged by downstream skills.
- [ ] Verify that the custom tracing rules are active on the observability side.
- [ ] Confirm the new multisig upgrades are active on the launch token repository.
- [ ] Conduct a review of the transaction simulation UX update.
